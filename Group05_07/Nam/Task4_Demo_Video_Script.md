# Kịch bản Demo Video (5 - 8 Phút)
**Chủ đề:** Tự động hóa kiểm thử API với AI và Postman

| Thời lượng | Thao tác trên màn hình | Lời thoại Audio / Thuyết minh |
| :--- | :--- | :--- |
| **0:00 - 0:45** (45s) | **Slide Intro**: Tiêu đề "Tự động hóa kiểm thử API với EShop".<br>Chuyển sang màn hình Terminal đang chạy Backend thành công (Log `Server is running on port 3000`). | Xin chào các bạn, hôm nay nhóm 05 sẽ demo cách ứng dụng AI Test Generation và Postman để kiểm thử tự động API cho hệ thống EShop. Như các bạn thấy, backend EShop của chúng ta đã được khởi chạy thành công ở localhost port 3000. |
| **0:45 - 2:30** (1m45s) | **Trình duyệt web**: Mở ChatGPT hoặc Claude. Copy một đoạn OpenAPI Specification (ví dụ API Checkout).<br>Gõ prompt: *"Generate a Postman test script for this Checkout API."*<br>Cuộn màn hình xem AI sinh code. Quét chọn đoạn code kiểm tra Schema rỗng (`additionalProperties: true`). | Đầu tiên, thay vì viết kịch bản test bằng tay mất thời gian, chúng ta sẽ nhờ AI sinh ra các Postman Test Scripts dựa trên OpenAPI Spec. Rất nhanh chóng, AI đã tạo ra code. Tuy nhiên, hãy nhìn vào đoạn Schema Validation này. AI định nghĩa một object rỗng. Đây là một Failure Mode cực kỳ phổ biến: Ảo giác dữ liệu, dẫn đến False Positive. Chúng ta sẽ cùng xem tác hại của nó trong Postman. |
| **2:30 - 4:00** (1m30s) | **Màn hình Postman**: Import đoạn code AI vừa sinh vào một request mới.<br>Cố tình đổi Request Body gửi đi thiếu trường `items`.<br>Nhấn **Send**. <br>Zoom vào phần **Test Results**: Hiện màu xanh PASS toàn bộ. | Mình đã đưa đoạn code của AI vào Postman. Bây giờ, mình sẽ cố tình gửi một request thiếu field bắt buộc là 'items' trong giỏ hàng. Nhấn Send... Các bạn có thể thấy API trả về lỗi, nhưng tab Test Results của Postman vẫn báo PASS xanh lè. AI đã bỏ qua Business Logic quan trọng! |
| **4:00 - 5:30** (1m30s) | **Màn hình Postman**: Sửa lại đoạn mã test. Thêm check `pm.expect(jsonData).to.have.property('items')`.<br>Chạy lại request -> Lần này Test báo FAIL (đúng như kỳ vọng). | Để khắc phục điểm mù này của AI, Tester con người cần can thiệp. Mình sẽ bổ sung logic kiểm tra các trường bắt buộc trực tiếp vào script. Chạy lại lần nữa... Lần này test đã chính xác báo FAIL. Đây chính là giá trị của Manual Edge Cases. |
| **5:30 - 7:00** (1m30s) | **Màn hình Postman**: Chọn thư mục EShop Collection v2.<br>Nhấn **Run Collection**.<br>Màn hình: Chọn thư mục EShop Collection v2. Bấm vào tab Variables để show cho khán giả thấy biến `{{token}}` đã được lấy tự động từ bước login. Sau đó nhấn Run Collection. Màn hình Runner chạy liên tục các API (Auth -> Product -> Cart -> Checkout).<br>Hiển thị màn hình Summary xanh/đỏ kết quả. | Cuối cùng, chúng ta sử dụng tính năng Collection Runner của Postman để chạy tự động luồng nghiệp vụ end-to-end. Điểm sáng ở đây là tính năng Chained-token Auth thông qua Collection Variables, ngay khi Đăng nhập thành công, token được lưu tự động để cấp quyền cho Giỏ hàng và Checkout phía sau mà không cần copy tay. Toàn bộ quá trình chỉ mất vài giây, rất phù hợp để tích hợp vào CI/CD pipeline sau này. |
| **7:00 - 7:30** (30s) | **Slide Outro**: Tóm tắt 3 ý: AI tiết kiệm thời gian, Human QA bắt Edge cases, Postman chạy tự động.<br>Lời cảm ơn. | Qua đoạn demo, nhóm muốn nhấn mạnh: AI giúp tăng tốc, nhưng tư duy của QA Engineer mới là yếu tố quyết định chất lượng kiểm thử. Cảm ơn các bạn đã theo dõi phần demo của nhóm! |
