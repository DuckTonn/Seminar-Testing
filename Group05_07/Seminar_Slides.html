Postman Failure Modes — điểm mù của tool
Các lỗi thường gặp do công cụ, dễ gây hiểu lầm cho người test.
Failure mode Mô tả
Sai kiểu dữ liệu API trả về String, nhưng logic test yêu cầu Number. Báo lỗi mơ hồ khiến QA lầm tưởng API sai nghiệp vụ.
Quên chọn Environment Gửi URL hoặc Token chứa chuỗi chưa được parse (VD: {{baseUrl}}), dẫn đến API sập nhưng mất thời gian debug sai hướng.
Silent Failure trong Script Viết sai cú pháp JavaScript, Postman bỏ qua toàn bộ script bên dưới nhưng UI không báo đỏ — vẫn hiện "Pass" (ảo giác Pass).
Bài học: Luôn kiểm tra kỹ biến môi trường và sử dụng linter / console.log trong Postman để debug script.

Integration Testing vs Contract Testing
Hiểu rõ vấn đề của Integration Test truyền thống và cách Contract Test (Pact) giải quyết.
Integration Testing (E2E)
Phức tạp & Chậm chạp
Cần môi trường thực (Database, APIs khác) chạy cùng lúc. Dễ xảy ra lỗi Flaky (chập chờn).
Tốn chi phí bảo trì
Khi một service thay đổi, toàn bộ chuỗi test có thể thất bại, rất khó debug lỗi nằm ở đâu.
Phản hồi chậm
Developer phải đợi CI/CD chạy xong toàn bộ hệ thống mới biết code mình có làm hỏng gì không.
Contract Testing (với Pact)
Nhanh & Độc lập
Consumer và Provider được test tách biệt thông qua "Bản hợp đồng" (Pact file). Không cần deploy cả 2.
Phát hiện lỗi sớm
Biết ngay lập tức nếu API thay đổi làm vỡ logic phía FE/Consumer ngay tại máy local.
Tập trung vào giao tiếp
Chỉ kiểm tra định dạng dữ liệu (Request/Response) truyền qua lại, bỏ qua logic nghiệp vụ rườm rà.

Cách Pact hoạt động: Consumer-Driven
Quy trình hướng Người tiêu dùng (Consumer-Driven Contract). Mọi thay đổi nên xuất phát từ nhu cầu của FE/Client.
Giai đoạn 1: Consumer (Ví dụ: Web UI)
1
Viết Unit Test
Định nghĩa mong đợi (Expectation) về Request và Response
2
Chạy Test & Tạo Mock
Pact tạo Mock Server trả về dữ liệu ảo cho FE
3
Sinh ra Contract (Pact File)
File JSON chứa toàn bộ quy tắc giao tiếp
Giai đoạn 2: Provider (Ví dụ: Order API)
4
Nhận Contract
Provider lấy file JSON (từ Broker hoặc File System)
5
Pact Replay
Pact giả lập Consumer gửi Request thực tới Provider
6
Xác thực & So sánh
Kiểm tra Response có khớp với hợp đồng hay không

Lưu ý khi sử dụng Pact (Best Practices)
Những điểm cần tránh để Contract Testing mang lại hiệu quả cao nhất.
Anti-pattern (Nên tránh) Giải thích & Cách khắc phục
Dùng Pact cho Functional Test Pact không dùng để test logic nghiệp vụ. Chỉ test định dạng dữ liệu, status code, headers. Đừng nhồi nhét mọi Test Case vào Pact.
Bỏ qua Provider States Thiếu trạng thái ban đầu khiến Provider test hay thất bại. VD: Request thông tin User nhưng DB chưa có User đó. Giải pháp: Định nghĩa rõ Provider State trong Contract.
Public API chưa xác định Client Pact tốt nhất cho nội bộ (Microservices, FE-BE chung công ty). Nếu build Public API cho bên thứ 3 vô danh, dùng OpenAPI/Swagger Testing thì phù hợp hơn.
Ghi nhớ: Contract Testing nằm giữa Unit Test và E2E Test. Hãy tối ưu để tăng tốc độ phát triển và CI/CD.

Vai trò của CI/CD trong API & Contract Testing
Đưa việc kiểm thử vào quy trình tự động hóa liên tục nhằm phát hiện lỗi sớm.
Vấn đề khi chạy thủ công
Dễ bỏ sót kiểm thử
Khi code thay đổi liên tục, việc chạy test thủ công dễ bị quên hoặc bỏ qua.
Không đồng bộ môi trường
Việc chạy test dưới máy cá nhân phụ thuộc vào cấu hình riêng, dễ phát sinh lỗi môi trường cục bộ.
Giải pháp từ CI/CD (GitHub Actions)
Tự động kích hoạt (Trigger)
Tự động chạy lại toàn bộ test suite (Newman v2 & Pact) ngay khi push code hoặc tạo Pull Request.
Chốt chặn chất lượng (Quality Gate)
Ngăn chặn gộp code lỗi hoặc phá vỡ hợp đồng (breaking changes) vào nhánh main trước khi deploy.

Báo cáo kiểm thử & Quản lý Artifacts
Đảm bảo việc lưu trữ minh chứng kiểm thử trực quan và dễ tiếp cận.
Báo cáo / Minh chứng Chi tiết định dạng & Nội dung
Newman HTML Report Sử dụng plugin htmlextra xuất báo cáo dạng HTML trực quan, chi tiết từng request, body response và assertion pass/fail.
Pact Verification Logs Ghi đè log đối sánh hợp đồng ra tệp tin văn bản pact-verification-report.txt phục vụ việc debug lỗi vi phạm hợp đồng (mismatch).

Lưu ý: Cấu hình bước xuất báo cáo với thuộc tính if: always() để đảm bảo luôn thu thập được log kể cả khi test fail.
